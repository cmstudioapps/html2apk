Modo por cima

O html2apk transforma uma pasta web em um app Android instalável. Ele usa Cordova para empacotar a WebView e injeta uma bridge própria para que JavaScript consiga conversar com recursos nativos do Android.

1. EntradaHTML, CSS, JS, assets e app.json.
2. PreparaçãoConfiguração normalizada, ambiente Android detectado e projeto temporário criado.
3. PonteScripts e plugin nativo entram no app gerado.
4. SaídaCordova compila e o APK/AAB aparece em dist.

Por fora, o usuário chama funções como notificar(), salvarArquivo(), baixarArquivo() e definirPapelParede(). Por dentro, a ferramenta normaliza argumentos, espera Cordova ficar pronto, chama cordova.exec, entra no Java Android, pede permissões quando necessário e devolve o resultado para uma Promise.

await toast("Olá");
await vibrar(250);
await aguardar(5000);
await notificar({ titulo: "Pedido", texto: "Atualizado" });
await salvarArquivo("perfil.json", { nome: "Ana" });
await baixarArquivo("https://site.com/manual.pdf", "manual.pdf");

O resumo técnico é este: a interface continua sendo HTML, CSS e JavaScript, mas o app passa a ter uma camada nativa para acessar recursos que uma página comum não acessaria sozinha.

Cérebro da aplicação

O cérebro do html2apk é uma sequência de transformações. Cada camada reduz a incerteza para que a próxima etapa trabalhe com dados mais previsíveis.

config do usuário
  -> normalização
  -> validação
  -> ambiente Android
  -> projeto Cordova temporário
  -> cópia web
  -> injeção runtime
  -> plugin nativo
  -> build Android
  -> artefato final

Neurônio 1: intenção

A ferramenta junta defaults, arquivo de configuração e overrides. O override vence o arquivo; o arquivo vence o default. O merge é profundo para objetos simples, então configurações como keystore podem ser completadas sem apagar o restante.

Neurônio 2: normalização

Valores humanos viram valores previsíveis. vertical vira portrait, horizontal vira landscape, AAB força release, aliases de OneSignal viram um campo único, tema automático vira themeMode: auto e permissões viram um array limpo.

Neurônio 3: validação

Antes de copiar qualquer coisa, a ferramenta confere se webRoot e entryFile ficam dentro do projeto. Isso impede que uma configuração inválida faça o build acessar arquivos fora da área esperada.

Neurônio 4: isolamento

O build acontece em um projeto temporário. Essa decisão protege a pasta web do usuário contra pastas Android pesadas, plugins, plataformas e cache. O projeto original fica mais limpo; o temporário concentra os arquivos gerados durante a compilação.

O html2apk não converte HTML em layout Android nativo. Ele empacota a WebView e instala uma ponte nativa, mantendo o comportamento previsível e alinhado ao modelo do Cordova.

Fluxo de build

O build é uma orquestração. Ele não é apenas uma chamada ao Cordova; antes disso a ferramenta cria o contexto inteiro que Cordova precisa.

Etapa Lógica Vantagem Limite
Ambiente Descobre Java, Gradle, Android SDK e ajusta PATH efetivo. Menos dependencia de terminal configurado manualmente. SDK e JDK ainda precisam existir.
Copia web Copia a pasta web ignorando lixo comum ou usa lista files. Evita mandar node_modules e builds antigos para o APK. Projetos fora do padrao podem precisar declarar arquivos.
Config XML Sintetiza manifest, permissões, splash, ícone, deep links e preferências. O usuário escreve JSON; o Cordova recebe XML. Manifest não concede runtime permission sozinho.
Runtime Injeta early bridge, console opcional e cordova.js de forma idempotente. O usuário não precisa editar HTML manualmente. HTML muito incomum pode escapar da heurística textual.
Artefato Procura APK/AAB recursivamente e prefere debug/release esperado. Resiste a variações de pasta de saída do Cordova. Se Cordova não gerar nada, falha de forma explícita.

Doctor

O doctor executa comandos reais e checa pastas reais. Ele testa Java, Javac, Gradle, Cordova, Android SDK, platform-tools, cmdline-tools, sdkmanager, build-tools, plataforma Android e permissão de escrita. Ele não gera APK; ele identifica requisitos ausentes antes do build.

USB debug

O teste USB lê adb devices, ignora emuladores, trata unauthorized, trata offline e procura um aparelho físico autorizado. Primeiro tenta cordova run android --device; se isso falhar, gera um APK debug, instala com adb install -r -d e abre com monkey.

Runtime e bridge

A bridge é a parte que transforma uma função JavaScript em comportamento Android. Ela também evita que o usuário precise lidar diretamente com o ciclo de inicialização do Cordova.

JavaScript do app
  -> função global html2apk
  -> normalização de argumentos
  -> espera cordova.exec
  -> action nativa
  -> Java Android
  -> CallbackContext
  -> Promise resolve/reject

Early bridge

A early bridge é injetada cedo. Ela cria as funções globais antes do Cordova estar totalmente pronto e espera o canal nativo por até 10 segundos. Se a ponte fica pronta, executa. Se não fica, falha com erro claro.

Plugin JS

O plugin JS vive dentro do modelo Cordova e chama cordova.exec diretamente. Ele mantém a mesma API pública da early bridge, com nomes em português e aliases em inglês.

Normalizadores

Antes de chegar ao Java, a bridge ajusta formatos: notificação string vira objeto, arquivo simples vira payload, download por base64 vira origem base64, papel de parede entende alvo/target, storage seguro entende chave e valor.

Runtime console

Quando ligado, o console runtime envolve console.log, erros globais, promises rejeitadas, fetch, XMLHttpRequest e funções html2apk conhecidas. Ele mostra chamadas, retornos, falhas, rede, botão de copiar console e botão de copiar apenas erros.

Java nativo

No Android, o plugin recebe uma action e decide qual subsistema executar. Essa action é a linguagem comum entre JavaScript e Java.

Dispatcher

Recebe action, args e callbackContext. Cada action mapeia para uma operação: notificação, câmera, arquivo, download, wallpaper, biometria, localização e mais.

Callbacks pendentes

Quando uma permissão interrompe o fluxo, a bridge guarda o callback e as opções. Depois que o Android responde, ela retoma a operação original.

Thread certa

Operações visuais usam UI thread. Downloads e operações pesadas usam thread pool para não travar a WebView.

Eventos

Eventos nativos voltam para o JS via CustomEvent, como html2apk:event e eventos específicos.

Notificações

Notificação imediata cria canal, monta NotificationCompat.Builder, adiciona PendingIntent de clique e ações. Notificação agendada usa AlarmManager, salva estado e pode ser reprogramada depois de boot. Loops calculam a próxima notificação depois de cada disparo.

Arquivos

Arquivos internos usam armazenamento do app, nome sanitizado, metadados e FileProvider para abrir/compartilhar. A validação bloqueia separadores, dois pontos e .. no nome.

Downloads

Downloads convergem para uma fonte interna com InputStream, tamanho, mimeType e tipo de origem. URL usa HttpURLConnection; base64 vira ByteArrayInputStream; URI usa ContentResolver; arquivo local usa FileInputStream. A cópia acontece em blocos de 8192 bytes e a notificação de progresso atualiza em intervalos controlados.

Wallpaper

Papel de parede decodifica bitmap e usa WallpaperManager. Imagem estática pode ir para tela inicial, bloqueio ou ambos. Video wallpaper e fundo de chamada são tratados como limites conhecidos, porque dependem de live wallpaper ou fabricante.

Permissões

Manifest e runtime permission são coisas diferentes. O html2apk declara o que precisa e, quando chega a hora, pede o que o Android exige em runtime.

função precisa de permissão
  -> já concedida?
      executa
  -> não concedida?
      guarda callback e opções
      pede permissão
        concedida -> continua
        negada -> retorna status/erro
        negada permanente -> abre configuração quando faz sentido

O sistema usa busy state para evitar duas requisições concorrentes da mesma família. Câmera, microfone, notificação, localização e downloads com notificação passam por esse cuidado.

Algumas permissões são especiais: overlay usa tela própria, alarme exato pode exigir configuração, notificação no Android 13+ exige POST_NOTIFICATIONS. O html2apk abre o caminho correto, mas o usuário final ainda decide.

Como chamar essas funções

A regra de uso é sempre a mesma: chame no JavaScript do app, use await, trate erro e lembre que no navegador comum essas funções nativas não existem. Elas ficam disponíveis no APK gerado pelo html2apk.

try {
  const resultado = await algumaFuncaoInterpretada();
  console.log(resultado);
} catch (erro) {
  console.error("Função nativa falhou", erro);
}
A bridge espera o Android ficar pronto. Mesmo assim, o melhor hábito é chamar recursos nativos depois que a tela inicial carregou ou em resposta a uma ação do usuário.

Bluetooth simples

aoConectarBT((dispositivo) => {
  console.log("Conectado", dispositivo.nome);
});

aoReceberDadosBT((dados) => {
  console.log("Recebido", dados);
});

aoDarErroBT((erro) => {
  console.log("Erro Bluetooth", erro.mensagem || erro.message);
});

const dispositivos = await procurarBT();

if (dispositivos[0]) {
  await conectarBT(dispositivos[0].id);
  await enviarBT({
    mensagem: "Olá por Bluetooth"
  });
}
O Bluetooth usa RFCOMM clássico com suporte nativo a salas P2P (1-para-N). O lado que recebe (Host) registra os eventos e passa a escutar; se vários aparelhos (Amigos) se conectarem, o Host segura todas as conexões simultaneamente sem derrubar ninguém. aoDarErroBT() recebe falhas de escuta ou conexão. O outro procura, conecta e envia.

Broadcast: A função enviarBT() manda os dados JSON simultaneamente para todos os dispositivos atualmente conectados a você! Para aparecer na busca, o aparelho precisa estar pareado ou visível nas configurações Bluetooth.

Downloads com progresso

const permInstalacao = await solicitarPermissaoInstalacao();

  await instalarAtualizacao("https://site.com/app.apk", {
    titulo: "Atualizando App",
    mensagem: "Fazendo download da nova versão..."
  });
}


await abrirArquivo("relatorio.pdf");

  mimeType: "image/png",

});

A função solicitarPermissaoInstalacao() retorna o seguinte objeto:

{
  "suportado": true,
  "supported": true,
  "solicitado": true,
  "requested": true,
  "permitido": false,
  "granted": false
}
  • suportado / supported: true se for Android 8.0 (Oreo) ou mais recente, que exige essa checagem manual.
  • permitido / granted: true se o usuário já marcou a chave permitindo a instalação ou se for um Android antigo sem restrição. false se o app ainda não pode instalar APKs.
  • solicitado / requested: true se o app acabou de abrir nativamente a tela de "Instalar apps desconhecidos" para o usuário.
Após solicitar, a tela abre e o usuário volta depois de marcar a opção. Logo após (ou quando permitido for verdadeiro), você pode chamar a função instalarAtualizacao() passando a URL do APK. O sistema irá fazer o download usando o PackageInstaller nativo. Você pode passar as opções titulo e mensagem para o balão bloqueante que impede o uso do app enquanto atualiza.
Por padrão, o download fica no armazenamento do app para funcionar com abrirArquivo() e compartilharArquivo(). Para imagem ou vídeo aparecer na galeria do Android, passe { galeria: true }; o retorno inclui publicUri quando a publicação for feita.

Guarda dados na memória RAM do Android que sobrevivem a F5 no HTML.

// Se a chave não existir, lerDaSessao retorna null (como no sessionStorage padrão)
const ausente = await lerDaSessao("nao-existe");

Desktop e laboratório

A interface desktop é uma camada de operação sobre o mesmo núcleo do CLI. Ela não deve duplicar regra crítica; ela deve chamar as APIs centrais e mostrar o estado de forma clara.

Processo principal

Abre diálogos, roda build, roda doctor, cria laboratório, chama USB debug, gerencia janela e acessa o sistema de arquivos.

Preload

Expõe uma API segura para o renderer. Isso mantém Node fora da interface web e reduz o risco de a UI expor acesso indevido ao sistema.

Renderer

Cuida de idioma, tema, formulário, editor, logs, aba de códigos interpretados, categorias e laboratório de funções.

Laboratório de funções

O botão "Testar funções" cria um app temporário com runtime console ligado e botões para experimentar as APIs nativas no celular. Ele exige USB porque o objetivo é observar comportamento real, com permissões reais e execução no Android.

Limites reais

A ferramenta é mais confiável quando assume seus limites. Recursos que dependem do Android, do fabricante ou de permissões especiais precisam ser documentados com clareza.

  • WebView continua sendo WebView: HTML/CSS/JS rodam dentro dela.
  • Permissão declarada no manifest não significa permissão concedida.
  • Fabricantes podem mudar telas de permissão, bateria, overlay e wallpaper.
  • Base64 gigante consome memória; para arquivos grandes, streaming é melhor.
  • Notificações no Android 13+ dependem de POST_NOTIFICATIONS.
  • Alarmes exatos podem exigir configuração manual.
  • Video wallpaper depende de live wallpaper; fundo de chamada depende do app/fabricante.
  • Apps abertos são limitados por privacidade do Android moderno.
  • OneSignal exige configuração externa e nunca deve receber chave secreta dentro do APK.
  • Keystore perdida não é recuperada pela ferramenta.
  • Ciclo de vida inteligente: O html2apk suprime automaticamente eventos app:pausado e app:voltou quando janelas nativas bloqueantes (como pedir PIN ou câmera) são abertas. Isso protege o seu app de falsos-positivos e loops infinitos.

Como mexer sem quebrar

Uma mudança boa respeita o fluxo inteiro. Função nova não é só botão, nem só Java, nem só documentação. Ela precisa atravessar a ponte toda.

função nova
  -> nome PT-BR
  -> alias inglês
  -> normalizador JS
  -> action Cordova
  -> implementação Java
  -> permissão manifest/runtime se precisar
  -> retorno JSON consistente
  -> runtime console se fizer sentido
  -> aba códigos interpretados
  -> laboratório USB se fizer sentido
  -> teste de consistência JS/Java
  -> documentação

Para arquivo, preserve sanitização e FileProvider. Para permissão, preserve callback pendente e busy state. Para build, preserve temporário, logs e limpeza segura. Para UI, preserve preload como fronteira.

O html2apk organiza a complexidade do Android para que o usuário consiga transformar um projeto web em app instalável sem implementar cada recurso nativo manualmente.
